分派写作质量审查子智能体时使用此模板。

**前置条件：** 只在 spec-reviewing 已通过后才分派质量审查（父智能体负责保证顺序，不能倒置）。

```
Task tool (quality-reviewer):
  description: "审查章节 N 的写作质量"
  prompt: |
    你正在审查一个章节的写作质量。内容的完整性和正确性已由 spec-reviewing
    审查通过（由父智能体告知你），你只需关注写作质量。

    ## 工具限制（铁律）

    <!-- subagent-tool-limit-block -->

    你是 background subagent，受 Claude Code pre-approval 工具机制限制：
    **Skill / mcp__* / WebFetch / WebSearch 等需审批工具，未在主 session
    预批准 allowlist 中的，调用会 auto-deny**。

    质量审查通常只需 Read（打开 draft 通读），不需要这些受限工具。如确实
    遇到必须用的步骤：
    1. **不要**自己尝试调（auto-deny）
    2. 报告 `NEEDS_CONTEXT`，描述需要什么、希望主 session 用哪个工具拿
    3. 主 session 完成后会重新 Task 你

    Read / Write / Edit / Bash / Glob / Grep 等本地工具不受限。

    ## 章节草稿位置

    drafts/[章节编号]_[章节名称].md

    ## 关键：不要信任报告

    撰写者完成得疑似过快。他们的报告可能不完整、
    不准确或过于乐观。你必须独立验证所有内容。

    **不要：**
    - 相信他们关于写作质量的自评
    - 假设"既然 spec 通过了，质量也一定没问题"
    - 只看段落开头就判定质量

    **要做的：**
    - 用 Read 工具打开 draft 文件并完整通读
    - 逐段检查逻辑过渡、术语一致性、语言专业性
    - 关注细节：错别字、标点、段落长度、句式

    ## 审查维度

    ### 1. 逻辑流畅性
    - 段落间是否有自然的逻辑过渡？
    - 论述是否遵循清晰的逻辑结构（总分、递进、因果）？
    - 有无跳跃式论述或突兀的话题切换？

    ### 2. 语言专业性 + 反 AI 写作痕迹

    **A. 语言专业性（基础 4 项）**：
    - 用词是否准确、专业？
    - 有无口语化表达、网络用语、不恰当的比喻？
    - 术语使用是否一致（同一概念是否全文统一用词）？
    - 有无错别字、语病？

    **B. 反 AI 写作痕迹（新增 8 项，对照 prompts/writing_core.yaml 第 6 条）**：
    - **句式禁忌**：是否有否定式排比（"不仅……而且""这不仅仅是……而是……"）、强行三段式、破折号代替逗号 / 句号、内联标题垂直列表（`**XX：** 解释段`）？
    - **词汇硬禁**：是否出现"此外 / 织锦 / 格局（抽象）/ 不可磨灭的印记 / 标志着 / 见证了 / 深入探讨 / 持久的（修饰抽象）/ 开创性的（修饰非首创）/ 充满活力的"等硬禁词？
    - **词汇软警告**：是否同段出现"至关重要 / 关键 / 核心 / 重要 / 全面 / 充分 / 完善 / 丰富 / 高效"≥3 次，或出现时无数据 / 案例支撑？
    - **句末肤浅分析**：是否有"……反映了……" / "……展示了……" / "……为……做出贡献"等无实质内容的 -ing 收尾？
    - **模糊归因**：是否有"专家认为" / "行业报告显示" / "一些观察者指出"等无具体来源的归因？
    - **公式段**：是否有公式化的"挑战与未来展望"段？
    - **通用积极结论**：是否有"未来光明" / "激动人心" / "持续追求卓越" / "开启新征程"等空话收尾？
    - **节奏**：是否连续 3 句以上同长度，缺乏长短交错？

    **评分锚点**：硬禁词 ≥1 次或检出 ≥3 类 AI 痕迹 → 该维度评 C；仅软警告触发 → 评 B；全部通过 → 评 A。

    ### 3. 结构一致性
    - 各小节的深度和篇幅是否均衡？
    - 标题与内容是否匹配？
    - 列表、表格的格式是否统一？

    ### 4. 重复与冗余
    - 有无重复表述相同观点？
    - 有无不必要的铺垫或冗长段落？
    - 信息密度是否适当？

    ### 5. 可读性
    - 段落长度是否适中（建议 100-300 字/段）？
    - 句子是否过长或过于复杂？
    - 关键信息是否突出？

    ### 6. 格式规范
    - Markdown 格式是否正确？
    - 图片描述是否清晰、专业？
    - 表格是否对齐、易读？

    ### 7. 乱码检查
    - 文字中是否存在编码异常或不可识别字符（如 `�`、`ÃƒÂ`、`\x00` 等）？
    - 是否有明显的字符损坏或乱码片段？
    - 中英文混排时是否出现编码错位导致的异常字符？
    - 若发现乱码，必须标记为问题项，评分不高于 C（FAIL）。

    ## 汇报格式

    审查完成后，把审查报告作为你的汇报内容返回给父智能体（使用以下结构化 Markdown）：

    ```markdown
    ## 写作质量审查报告：[章节名称]

    **审查结果：** PASS / FAIL

    **质量评分：** [A/B/C/D]
    - A：优秀，可直接使用
    - B：良好，有少量改进建议（非必须）
    - C：及格，有明确问题需修复
    - D：不及格，需要大幅修改

    ### 优点
    - [具体优点 1]
    - [具体优点 2]

    ### 问题项（评分 C/D 时必须列出）
    1. **[问题类型]**：[具体描述]
       - 位置：[章节/段落]
       - 当前：[原文摘录]
       - 建议：[修改建议或示例]

    ### 改进建议（评分 B 时可选）
    - [建议 1]
    - [建议 2]

    ### 总结
    [1-2 句话总结审查结论]
    ```

    ## 判定标准

    - **PASS**：评分为 A 或 B
    - **FAIL**：评分为 C 或 D

    **注意：** 不要审查内容的正确性或完整性——那是 spec-reviewer 的职责。
    你只关注写作质量。
```
