# 编写技能

## 概述

**编写技能就是将测试驱动开发应用于流程文档。**

你编写测试用例（带子智能体的压力场景），观察它们失败（基线行为），编写技能（SKILL.md），观察测试通过（智能体遵守规则），然后重构（堵住漏洞）。

**核心原则：** 如果你没有观察到智能体在没有该技能时失败，你就不知道这个技能是否教了正确的东西。

## 技能类型

| 类型 | 例子 | 测试方式 |
|------|------|---------|
| 纪律执行类 | 审查规则、流程约束 | 压力场景：时间 + 沉没成本 + 疲惫 |
| 技术操作类 | 知识检索、配图生成 | 应用场景 + 边界情况 + 缺失信息 |
| 模式思维类 | 苏格拉底提问、结构化分析 | 识别 + 应用 + 反例 |
| 参考文档类 | API 文档、格式规范 | 检索 + 应用 + 覆盖度 |

## SKILL.md 结构

```yaml
---
name: 技能名称-用连字符
description: 触发条件描述（何时使用，不是做什么）
---
```

**关键：description = 何时使用，不是技能做什么**

当 description 总结了技能的工作流时，Claude 可能跟随描述而跳过阅读完整技能内容。description 只写触发条件。

```yaml
# 错误：总结了工作流
description: 每个任务分派子智能体并在任务间进行审查

# 正确：只有触发条件
description: 当在当前会话中执行包含独立任务的撰写计划时使用
```

## 目录结构

```
skills/
  技能名称/
    SKILL.md              # 主文档（必需）
    prompts/              # 提示词文件（按需）
    scripts/              # 工具脚本（按需）
```

**扁平命名空间** — 所有技能在一个可搜索的空间中。

## 命名规则

- 使用主动动词：`brainstorming` 而非 `brainstorm-skill`
- 用连字符分隔：`knowledge-retrieval` 而非 `knowledgeRetrieval`
- 描述你做的事：`spec-reviewing` 而非 `review-tool`
- 动名词（-ing）适合描述流程

## 红-绿-重构（技能的 TDD）

### 红：编写失败的测试（基线）

在没有技能的情况下运行压力场景，逐字记录：
- 智能体做了什么选择？
- 使用了什么合理化借口？
- 哪些压力触发了违规？

### 绿：编写最小技能

针对那些具体合理化借口编写技能。不要为假设情况添加额外内容。

### 重构：堵住漏洞

智能体找到新的合理化借口？添加明确反驳。重新测试直到无懈可击。

## 铁律

```
没有失败的测试就不写技能
```

先写技能再测试？删掉它。重新开始。

**无例外：**
- 不适用于"简单的添加"
- 不适用于"只是加一个章节"
- 不适用于"文档更新"

## 常见合理化借口

| 借口 | 现实 |
|------|------|
| "技能显然很清晰" | 对你清晰 ≠ 对其他智能体清晰 |
| "这只是参考资料" | 参考资料可能有遗漏。测试检索。 |
| "测试太过了" | 未测试的技能总有问题 |
| "没时间测试" | 部署未测试的技能浪费更多时间 |
