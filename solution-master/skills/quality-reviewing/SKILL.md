# 写作质量审查

通过分派独立的审查子智能体，审查撰写产出的写作质量、结构一致性和专业性。

**核心原则：** 内容正确性已验证，现在聚焦于写作质量。审查子智能体不重复检查内容完整性。

**前置条件：** spec-reviewing 必须已通过。未通过则不可执行本审查。

## 流程

1. **确认 spec-reviewing 已通过**（由父智能体在会话里保证顺序）
2. **分派 quality-reviewer 子智能体**（`agents/quality-reviewer.md`）
3. 传入：实际撰写产出文件路径
4. 等待审查结果
5. **将审查报告展示给用户**：显示质量评分（A/B/C/D）和关键发现（优点 + 问题项）

## 判定标准

- **PASS**（评分 A/B）：任务完成 → 进入下一个任务
- **FAIL**（评分 C/D）：撰写子智能体修复 → 重新质量审查

## 审查维度

| 维度 | 检查内容 |
|------|---------|
| 逻辑流畅性 | 段落间逻辑过渡？论述结构清晰？ |
| 语言专业性 | 用词准确专业？术语一致？无错别字？ |
| 结构一致性 | 各节深度均衡？标题与内容匹配？ |
| 重复与冗余 | 无重复观点？信息密度适当？ |
| 可读性 | 段落长度适中？句子不过长？ |
| 格式规范 | Markdown 格式正确？表格对齐？ |
| 乱码检查 | 文字中是否存在编码异常、不可识别字符或明显的字符损坏？ |

## 质量评分

| 等级 | 含义 | 结果 |
|------|------|------|
| A | 优秀，可直接使用 | PASS |
| B | 良好，有少量改进建议（非必须） | PASS |
| C | 及格，有明确问题需修复 | FAIL |
| D | 不及格，需大幅修改 | FAIL |

## 铁律

- **必须在 spec-reviewing 通过后才能执行**——顺序不可颠倒
- **审查必须由独立子智能体执行**——撰写者不能审查自己
- **FAIL 必须修复后重新审查**——不可跳过
