
# 内容正确性审查

通过分派独立的审查子智能体，验证撰写产出是否完整、准确地满足计划要求。

**核心原则：** 不要信任撰写者的报告——审查子智能体必须亲自阅读实际产出并逐条比对计划要求。

## 流程

1. **分派 spec-reviewer 子智能体**（`agents/spec-reviewer.md`）
2. 传入：
   - 任务计划（要点清单、验收标准、目标字数、配图需求）
   - 实际撰写产出文件路径
3. 等待审查结果
4. **将审查报告展示给用户**：PASS 时显示通过确认；FAIL 时显示问题列表和修复建议

## 判定标准

- **PASS**：所有要求项均已满足，无禁止项 → 进入 quality-reviewing
- **FAIL**：存在未满足的要求项 → 撰写子智能体修复 → 重新审查

## 审查维度

| 维度 | 检查内容 |
|------|---------|
| 完整性 | 每个要点都有对应内容？字数达到目标？（只能多不能少，超标不算问题） |
| 准确性 | 正确理解计划意图？技术描述准确？ |
| 合规性 | 来源标注？配图插入？标题层级正确？ |
| 禁止项 | 无 TODO/TBD？无自相矛盾？无未标注来源的数据？ |

## 铁律

- **审查必须由独立子智能体执行**——撰写者不能审查自己
- **必须在 quality-reviewing 之前通过**——顺序不可颠倒
- **FAIL 必须修复后重新审查**——不可跳过
