# TAA — 招标分析助手 技术文档

> 版本：v2.4.0 | 角色：分析者 (Tender Analysis Assistant) | 服务对象：乙方（投标方）

---

## 1. 设计目标

TAA 是乙方收到招标文件后的**第一个处理环节**，负责深度分析招标文件并生成两个核心产出物：

1. **招标分析报告**（Markdown）：7 个分析模块 + 可选行业扩展模块
2. **投标文件大纲**（DOCX）：包含 Part A 大纲结构 + Part B 逐节写作指导 + Part C 需求-章节对应表

---

## 2. M1-M7 七模块分析框架

TAA 的核心是 7 个分析模块，定义在 `prompts/analysis.yaml` 中：

### M1：项目概况与投标建议

**分析内容**：
- **基本信息表**：项目名称、招标编号、采购单位、行业领域、预算金额、交付周期、投标截止时间、核心采购内容
- **业务匹配度分析**：✅ 完全匹配项 / ⚠️ 需定制项 / ❌ 能力缺口项
- **投标建议**：四级建议（强烈建议/建议/谨慎/不建议），每项需 3-5 条具体依据

**数据来源**：招标文件正文 + 产品能力说明书（如提供）

### M2：技术与服务要求分析矩阵

**这是 TAA 最核心的模块**，后续所有模块和大纲生成都依赖 M2 的输出。

**分析内容**：将招标文件中的所有技术与服务要求逐条提取为结构化矩阵：

| 字段 | 说明 |
|------|------|
| 需求编号 | T1, T2, ... |
| 分类 | 技术能力类（动态分类）或服务管理类（6 个固定分类） |
| 要求原文 | 招标文件中的完整要求描述 |
| 出处 | 第X章 P.XX |
| 优先级 | 【必须】/【必须/废标】/ 普通 |
| 标记 | ★ / * / 空（基于标记映射表判定） |
| 厂商支持度 | 完全支持 / 需定制 / 风险-需评估 / 不支持 |
| 匹配依据 | 产品能力索引条目 ID 或 AI 模糊评估标注 |

**分类体系**：
- **技术能力类**：参考分类 + 动态扩展（不局限于预定义分类）
  - 参考分类：容器管理、集群管理、网络管理、存储管理、安全管理、监控告警、CI/CD、微服务治理、AI/GPU
- **服务管理类**：6 个固定分类必须全覆盖
  - 实施部署、培训服务、售后维保、技术支持、SLA 服务、质量保障

**提取范围**：覆盖全格式（表格/段落/列表/附件），不因格式差异遗漏。

**支持度判定**：
- 有 `--product` 参数 → 基于产品能力索引的结构化匹配（精确匹配 → 模糊匹配 → 语义匹配三级）
- 使用 AnythingLLM → 对每条技术要求调用语义检索（score >= 0.65 视为匹配）
- 无产品数据 → AI 模糊评估（基于互联网公开信息）

### M3：商务要求与合同条款分析

**分析内容**：
- 资质要求（营业执照/认证/业绩）
- 保证金要求
- 付款条件与风险
- 合同关键条款（违约/知识产权/保密）
- 投标文件格式要求

### M4：评分细则与得分策略

**分析内容**：

| 字段 | 说明 |
|------|------|
| 评分项 ID | S1, S2, ... |
| 评分项名称 | 从评分标准中提取 |
| 最高分值 | 该项满分 |
| 分类 | 技术方案/实施服务/安全合规/... |
| 评分规则 | 具体得分区间和条件 |
| 厂商优势 | 基于 M2 支持度判断 |
| 拿高分策略 | 针对性的写作和响应建议 |

**特别关注**：≥10 分的高分值项标记为重点展开项，供 taw 写作时优先处理。

### M5：废标条款与风险排序

**分析内容**：
- **废标条款**：从全文档（含附件和合同范本）提取所有可能导致废标的条件
- **扣分条款**：可能导致扣分的技术/商务条件
- **风险排序**：按影响程度排序（致命→严重→一般→轻微）

**检测范围**：必须覆盖全文档所有章节，包括附件和合同范本。

### M6：标书格式要求 Checklist

**分析内容**：
- 投标文件组成清单
- 格式要求（字体/纸张/装订/密封）
- 签章要求
- 份数要求
- 电子文件要求

### M7：标书撰写指导

**分析内容**：
- **7.1 撰写优先级**：P1（高优先级）→ P2 → P3（低优先级）
- **7.2 关键词清单**：每章需覆盖的技术/商务/合规关键词
- **7.3 评分点写法建议**：具体到每个评分项的响应策略
- **7.4 差异化亮点**：在哪些章节展现厂商独特优势
- **7.5 红线**：⛔ 红线（绝对不可触碰）和 ⚠️ 慎重项

**7.6 跨模块交叉验证**：M7 最后执行验证，检查 M2/M5 是否有遗漏，发现遗漏需回溯补充。

## 3. 行业扩展模块

根据 Phase 0.2 自动判断的行业类型，在 M1 之后插入 E1-E3 扩展模块：

### 政府IT 扩展

| 模块 | 内容 |
|------|------|
| E1 预算合规分析 | 预算金额合理性、资金来源、最高限价、付款方式 |
| E2 政府采购合规分析 | 采购方式、中小企业政策、进口产品限制、时间要求 |
| E3 安全保密分析 | 等保级别、数据本地化、安全产品国产化 |

### 国企/央企 扩展

| 模块 | 内容 |
|------|------|
| E1 国资监管合规 | 采购管理、投资决策程序、信息公开 |
| E2 信创国产化要求 | CPU/OS/数据库/中间件型号要求、信创认证 |
| E3 廉政风险防控 | 廉洁从业、回避制度、审计监督 |

### 金融扩展

| 模块 | 内容 |
|------|------|
| E1 金融监管合规 | 银保监会/证监会/央行监管要求 |
| E2 数据安全与隐私 | 金融数据分级、个人金融信息保护 |
| E3 业务连续性 | RTO/RPO 要求、高可用架构、灾备体系 |

**行业判断规则**：统计各行业关键词在招标文件中的出现次数，选择最多的行业类型；相同时按优先级：政府IT > 金融 > 国企 > 通用。

## 4. 产品能力来源决策

TAA 支持多种产品能力数据来源，按优先级降级：

```
AnythingLLM（语义检索）
    ↓ 不可用或结果不足
本地 V2 三层索引（L0/L1/L2）
    ↓ 不存在
本地 V1 索引（product.yaml）
    ↓ 不存在
AI 模糊评估（无产品数据）
```

### V2 三层索引架构

由 `tools/indexer_v2.py` 生成，按语义聚类为 11 个分类：

| 层级 | 文件 | 大小 | 内容 |
|------|------|------|------|
| L0 | `product_l0.yaml` | ~2KB | 11 个分类名 + 条目数 |
| L1 | `product_l1_*.yaml` × 11 | 0.5-19KB | 条目摘要（name + keywords） |
| L2 | `product_l2_*.yaml` × 11 | 1-35KB | 完整详情（全字段） |

**按需加载策略**：Phase 1 中根据 M2 需求分类，仅加载相关分类的 L1/L2 索引，节省 70-90% token。

### AnythingLLM 查询机制

对 M2 每条技术要求：
1. 提取核心关键词，构建查询语句
2. 调用 `anythingllm_search(query, workspace)` 
3. 取 score >= 0.65 的结果
4. 查询缓存避免重复调用
5. 失败降级为 AI 模糊评估

## 5. 工作流程（3 Phase）

### Phase 0：文件预处理与行业判断

```
0.0 参数解析（读取 prompts/phase0_params.md）
    ├── 帮助参数 → 输出帮助退出
    ├── --vendor → 设置 VENDOR_NAME
    ├── --product → 解析产品能力文件（Excel/Markdown）
    ├── --build-index → 构建索引退出
    ├── --kb-source → 产品能力来源策略
    └── --anythingllm-workspace → 指定 workspace

0.1 文件确认（读取上传文件，逐一列出）
0.2 行业判断（关键词统计 → 加载对应模板）
0.3 启动确认（输出摘要后直接进入 Phase 1）
```

**输出目录**：`./output/`

### Phase 1：招标分析报告

```
Step 1: 准备上下文
    ├── 读取 prompts/analysis.yaml
    ├── {VENDOR_NAME} 变量替换
    └── 产品能力上下文准备（AnythingLLM / 本地索引 / AI 模糊）

Step 2: 加载行业模板

Step 2.5: 标记映射预扫描
    └── 全文扫描，识别 ★、*、△ 等特殊标记的语义定义

Step 3: 执行 7 模块分析
    └── M1 → [E1-E3] → M2 → M3 → M4 → M5 → M6 → M7（含 7.6 交叉验证）
```

**输出**：`招标分析报告_<timestamp>.md` + `phase1_handoff.yaml`（结构化交接文件）

### Phase 1.5：匹配验证（条件执行）

**执行条件**：仅当提供了 `--product` 参数时。

对 M2 中标记为 ⚠️ 的条目进行二次验证：
1. 同义词扩展检索
2. 跨分类检索
3. 模糊语义匹配

目的：减少误报，提高支持度判断准确性。

### Phase 2：投标文件大纲（子 Agent 执行）

**关键设计**：Phase 2 通过 Agent 工具启动独立子 agent 执行，获得全新 context 窗口，避免继承 Phase 1 累积的长对话历史。

子 Agent 读取的资源：
- `prompts/phase2_instructions.md`：完整执行指令
- `prompts/outline.yaml`：大纲格式规范
- `phase1_handoff.yaml`：Phase 1 结构化交接数据
- `tools/docx_outline_template.py`：DOCX 生成模板

**大纲三部分结构**：

#### Part A：大纲目录结构

```
一、技术部分
  1.1 技术偏离表
  1.2 项目理解与需求分析
  1.3 总体方案设计
  1.4 [动态专项响应章节]     ← 基于 M2 分类自动生成
  1.5 实施方案与计划
  1.6 质量保障方案
  1.7 安全方案
  1.8 国产化适配方案（如有信创要求）
  1.9 项目团队与人员配置
  1.10 售后服务方案
    1.10.1 质保服务方案
    1.10.2 运维服务方案
    1.10.3 服务响应承诺
    1.10.4 增值服务方案
  1.11 培训方案
```

**仅保留技术部分**（v2.1.0+），删除商务、报价、附件等章节。编号从 `1.X` 开始。

#### Part B：逐节写作指导

每个章节的写作要求：目标字数、重点内容、需覆盖的评分项、参考素材来源。

#### Part C：需求-章节对应表

将 M2 的每条技术要求映射到大纲的具体章节，确保所有【必须】条款在大纲中有响应位置。

### Phase 3：任务完成与清理

输出确认 + 自动清理临时目录。

## 6. 交接文件 phase1_handoff.yaml

这是 Phase 1 → Phase 2 的**结构化数据桥梁**，避免 Phase 2 重新解析完整报告：

```yaml
generated_at: "ISO 时间戳"
report_file: "Phase 1 报告路径"
project_name: "项目名称"
vendor_name: "厂商名称"
industry_type: "行业类型"

m2_requirements:     # M2 全部条目（保留完整原文）
m2_categories:       # M2 分类汇总（用于大纲结构决策）
m4_scoring:          # M4 评分要点（≥10分项完整）
m5_risks:            # M5 废标与风险（完整列出）
m7_priorities:       # M7 写作优先级
industry_extensions: # 行业扩展模块摘要
marker_mapping:      # 标记映射表
```

## 7. Prompt 文件架构

```
skills/taa/
├── SKILL.md                       # 主编排 Prompt（Phase 0-1 + 子 Agent 调度）
├── prompts/
│   ├── phase0_params.md           # Phase 0 参数解析详细规则（外置减少主编排体积）
│   ├── analysis.yaml              # 7 模块分析框架（M1-M7 的 prompt 定义）
│   ├── outline.yaml               # 大纲格式规范（Part A/B/C 结构）
│   └── phase2_instructions.md     # Phase 2 子 Agent 完整执行指令
├── tools/
│   ├── indexer_v2.py              # V2 三层索引生成器
│   └── docx_outline_template.py   # DOCX 大纲生成模板（字体/样式/结构）
└── templates/
    └── builtin/
        ├── government_it.yaml     # 政府IT 行业模板（含 E1-E3）
        ├── finance.yaml           # 金融行业模板
        ├── state_owned_enterprise.yaml  # 国企模板
        └── general.yaml           # 通用模板
```

## 8. 与其他 Skill 的关系

```
招标文件 → [TAA] → 招标分析报告.md + 投标文件大纲.docx
                          │                    │
                          │                    └──→ [TAW] 读取大纲结构
                          └──────────────────────→ [TAW] 读取 M2/M4/M5/M7
                          └──────────────────────→ [TRV] 审核（analysis/outline 类型）
```

TAA 的两个产出物是 TAW 撰写标书的**唯一输入**。TAW 通过读取分析报告的 M2（技术要求）、M4（评分标准）、M5（废标条款）、M7（撰写指导）获取撰写上下文，通过读取大纲确定章节结构和子节标题。
