# TPL — 招标技术规格与评标办法生成 技术文档

> 版本：v2.0 | 角色：策划者 (Tender Planner) | 服务对象：甲方（招标方）

---

## 1. 设计目标

TPL 的核心使命是**反控标转换**：将乙方的产品功能清单，以自然、无控标痕迹的方式转换为甲方视角的招标技术规格与评标办法。

**"反控标"** 是指消除招标文件中可能被认为是为特定供应商量身定制的痕迹，使招标文件看起来是基于通用行业需求编写的。

---

## 2. 核心能力：反控标转换

反控标转换是 TPL v2.0 的核心，**全部通过提示词驱动 AI 模型执行**，不硬编码映射规则。转换规则定义在 `prompts/anti_control.yaml` 中。

### 2.1 五步转换流程

| 步骤 | 转换维度 | 具体操作 | 示例 |
|------|---------|---------|------|
| 1 | 术语中性化 | 去除品牌名、产品名、公司名、专有特性名 | "ACP 容器平台" → "容器管理平台" |
| 2 | 指标合理化 | 使用下限表达，数值浮动 70%-85% | "支持 10000 Pod" → "不低于 7500 个容器实例" |
| 3 | 需求泛化 | 实现方案 → 功能结果描述 | "基于 Istio 的服务网格" → "应具备服务网格治理能力或同等方案" |
| 4 | 分布均衡 | 多技术域覆盖，等级比例分配 | 必须/应/宜 ≈ 60-75%/15-25%/5-15% |
| 5 | 自检 | 品牌残留零容忍 + 集中度检查 | 单一域不超 40%，无品牌残留 |

### 2.2 控标点机制

产品功能描述中包含"控标点"字样的条目享受特殊处理：
- **必定纳入**招标要求，不受 level 压缩比影响
- **保留核心技术指标原值**（不降级指标），但仍执行术语中性化
- 评标办法中**优先映射为评分项**

## 3. 输入与参数

### 3.1 输入来源（三种模式）

| 模式 | 参数 | 描述 |
|------|------|------|
| 文件输入 | `<file>` | .txt/.md/.xlsx/.pdf 格式的产品功能清单 |
| 知识库索引 | `--kb` | 使用 `Local-KnowledgeBase/.index/product_l*.yaml` 三层索引 |
| 组合输入 | `<file> --kb` | 知识库为基础，文件为补充/覆盖 |

### 3.2 四级细致程度

| 级别 | 要求条目 | 页数 | 适用场景 | KB 加载深度 |
|------|---------|------|---------|------------|
| `detailed` | 40-60 条 | 15-20 页 | 大型项目 >500 万 | 全部 L2 详情 |
| `standard` | 15-25 条 | 8-12 页 | 中型项目 100-500 万 | Top 3 分类的 L2 |
| `general` | 8-12 条 | 4-6 页 | 小型项目 50-100 万 | 仅 L1 摘要 |
| `brief` | 5-8 条 | 2-3 页 | <50 万/内部参考 | 仅 L0 概述 |

### 3.3 四大行业模板

每个模板定义在 `templates/<industry>.yaml` 中，包含行业特征、技术规格框架、评标默认配置和反控标覆盖。

| 模板 | 核心关注点 | 价格权重 | 技术权重 | 专项章节 |
|------|-----------|---------|---------|---------|
| `government` | 合规性、信创、等保 | 30% | 50% | 等保安全、国产化/信创 |
| `finance` | 金融级安全、高可用 | 25% | 50% | 高可用架构、金融安全、监管合规 |
| `soe` | 自主可控、稳定性 | 30% | 45% | 自主可控、国产化适配、长期支持 |
| `enterprise` | 性价比、灵活性 | 40% | 40% | 灵活部署、快速交付、成本优化 |

## 4. 工作流程（5 Phase）

### Phase 0：参数解析与输入加载

```
Step 1: 帮助参数检测 → -h/--help 输出帮助退出
Step 2: 解析参数（功能清单/--kb/--project/--template/--level/--no-scoring）
Step 3: 参数验证（输入源/文件存在性/模板有效性/level 有效性）
Step 4: 宣告参数摘要
```

**输出目录**：`./output/tpl/`

**配置加载**：读取 `tw_config.py` 获取 `tpl.default_template` 和 `tpl.default_level` 作为默认值。

**输入加载**：
- 文件输入：读取文件，按格式解析功能条目
- KB 索引：按 level 分层加载 L0/L1/L2 三层索引
- 项目概述：提取项目名称、背景、预算、工期等
- 行业模板：读取 `templates/<industry>.yaml`

### Phase 1：产品功能分析

对所有输入的功能条目执行：

1. **控标点识别**：扫描描述中是否包含"控标点"三字 → 标记 `is_control_point: true`
2. **功能分类**：按技术域分 8 类（功能/性能/安全/集成/可靠性/交付/实施/服务）
3. **量化指标提取**：识别数值指标（"10000 Pod"、"99.99%"、"P99 < 50ms"）
4. **功能分级**：核心/标准/高级/可选

### Phase 2：反控标转换（核心 Phase）

读取 `prompts/anti_control.yaml` 和 `prompts/level_rules.yaml`。

```
2.1 控标点条目预处理
    ↓ 提取 is_control_point=true 的条目，保留核心指标
2.2 术语中性化
    ↓ 去品牌，用行业通用术语
2.3 指标合理化
    ↓ 下限表达，浮动 70%-85%（控标点保持原值）
2.4 需求泛化
    ↓ 功能结果导向，加"或同等"表述
2.5 分布均衡
    ↓ 多域覆盖，必须/应/宜比例
2.6 反控标自检
    ↓ 品牌残留（零容忍）+ 集中度 + 等级比例 + 自然度
```

### Phase 3：技术规格生成

读取 `prompts/technical.yaml`，将 Phase 2 转换后的要求组织为技术规格章节：

**标准章节结构**（10 章）：
1. 项目总体说明
2. 功能需求规格（核心，反控标转换主应用区）
3. 性能需求规格
4. 接口与集成要求
5. 安全要求
6. **[行业专项章节]**（动态注入，来自行业模板）
7. 质量与可靠性要求
8. 交付物要求
9. 实施与服务要求
10. 验收标准

**Level 控制**：
- `detailed`：逐项展开，每条附验收标准
- `standard`：核心详细，次要归组
- `general`：按类概括
- `brief`：扁平列表

### Phase 4：评标办法生成

**执行条件**：未指定 `--no-scoring` 时执行。

读取 `prompts/scoring.yaml`。

```
4.1 确定评标方法和权重（来自行业模板）
4.2 技术评分维度映射（Phase 3 章节 → 评分维度）
    ├── 量化要求 → 客观评分项
    ├── 描述性要求 → 主观评分项（优/良/一般/较差四档）
    └── 控标点维度 → 分值适当提高
4.3 商务评分（来自行业模板 scoring_defaults）
4.4 价格评分（来自 scoring.yaml 公式选项）
4.5 无效投标认定（从"必须"要求提取 5-8 条）
4.6 交叉检查（必须项↔评分项对应、控标点映射完整）
```

**维度数量按 level**：detailed 6-8 个 / standard 4-6 个 / general 3-4 个 / brief 2-3 个

### Phase 5：DOCX 输出与交付

1. **DOCX 生成**：使用 Python + python-docx，复用 taa 的字体设置函数
2. **反控标最终检查**：品牌名最终扫描，确保零残留
3. **文件交付**：输出到 `./output/tpl/`
4. **输出确认**：包含技术要求条目数、控标点数、评标维度数

## 5. Prompt 文件架构

```
skills/tpl/
├── SKILL.md                    # 主编排 Prompt
├── prompts/
│   ├── anti_control.yaml       # 反控标转换规则（术语/指标/需求/分布/自检）
│   ├── level_rules.yaml        # 四级细致程度定义和压缩比
│   ├── technical.yaml          # 技术规格章节结构和写作规则
│   └── scoring.yaml            # 评标办法结构和生成规则
└── templates/
    ├── government.yaml         # 政府行业模板
    ├── finance.yaml            # 金融行业模板
    ├── soe.yaml                # 央国企模板
    └── enterprise.yaml         # 通用企业模板
```

## 6. 与其他 Skill 的关系

```
产品功能清单 → [TPL] → 招标技术规格 + 评标办法
                              ↓
                         [TRV] 审核（tender_doc 类型）
                              ↓
                         甲方发布招标文件
                              ↓
                    乙方获取招标文件 → [TAA] 分析
```

TPL 的输出可直接交给 TRV 进行 `--type tender_doc` 审核，检查反控标合规性和评标办法合理性。
