# TRV — 审核者 技术文档

> 版本：v1.5.0 | 角色：审核者 (Tender Reviewer) | 服务对象：甲方 + 乙方

---

## 1. 设计目标

TRV 是四角色系统中的**质量把控角色**，对各阶段产出物进行多维度审核，识别问题与风险，提供改进建议。

TRV 覆盖招投标全流程的五种审核场景：

| 审核类型 | 审核对象 | 产出来源 | 核心检查点 |
|---------|---------|---------|-----------|
| `tender_doc` | 技术规格与评标办法 | TPL | 反控标合规性、评标办法合理性 |
| `analysis` | 招标分析报告 | TAA | 分析完整性、数据准确性 |
| `outline` | 投标文件大纲 | TAA | 结构完整性、需求覆盖度 |
| `chapter` | 章节草稿 | TAW | 内容质量、评分契合度 |
| `full_bid` | 完整技术标书 | TAW（合并） | 整体一致性、风险检查 |

---

## 2. 四维审核框架

TRV 对每种审核类型执行最多 4 个维度的检查，每个维度有独立的 Prompt 配置文件：

### 2.1 完整性检查（Completeness）

**配置文件**：`prompts/completeness.yaml`

**检查逻辑**：根据 `--type` 参数加载对应的检查清单，逐项检查文档是否包含必备要素。

**检查结果字段**：

| 字段 | 说明 |
|------|------|
| 检查项 | 应包含的文档要素 |
| 状态 | ✅ 存在 / ❌ 缺失 |
| 严重程度 | critical（严重）/ major（一般）/ minor（建议） |
| 说明 | 具体缺失描述或补充建议 |

**按 --level 过滤**：
- `critical`：仅显示 critical 级别
- `high`：显示 critical + major
- `all`：显示所有级别

**各审核类型检查清单**：

- **tender_doc**：9 章技术规格 + 6 章评标办法（v1.2.0 对齐 tpl v2.0）
- **analysis**：M1-M7 各模块完整性
- **outline**：技术部分章节结构完整性、服务管理章节覆盖（v1.3.0 适配大纲简化）
- **chapter**：标题层级、内容深度、格式规范
- **full_bid**：技术标书整体结构完整性（v1.4.0 支持分块审核，检查项区分 `scope: chapter` 和 `scope: global`）

### 2.2 合规性审查（Compliance）

**配置文件**：`prompts/compliance.yaml`

**检查逻辑**：检查文档是否符合相关法律法规和行业规范。

**检查结果字段**：

| 字段 | 说明 |
|------|------|
| 审查项 | 法规或规范要求 |
| 合规状态 | 合规 / 不合规 / 待确认 |
| 风险等级 | high / medium / low |
| 法规引用 | 具体法条或标准编号 |
| 修改建议 | 如何修正使之合规 |

**各审核类型的合规重点**：

- **tender_doc**：反控标合规性（品牌残留、指标分布均衡性）、评标办法合规性（权重合理性、评分标准与技术规格对应关系）
- **analysis**：数据准确性、出处标注完整性
- **outline**：需求覆盖完整性、章节逻辑合理性
- **chapter**：技术描述准确性、承诺合理性
- **full_bid**：全文一致性、格式规范（v1.4.0 分块模式下由跨章节一致性检查 Agent 执行）

### 2.3 评分契合度分析（Scoring Alignment）

**配置文件**：`prompts/scoring_alignment.yaml`

**检查逻辑**：将文档内容与评分标准进行映射，评估每个评分点的覆盖程度。

**支持结构化输入**（v1.3.0）：scoring_alignment 支持 `phase1_handoff.yaml` 作为可选输入，提高分析精度。

**检查结果字段**：

| 字段 | 说明 |
|------|------|
| 评分项 | 评分标准中的具体项 |
| 分值 | 该项满分 |
| 覆盖程度 | 完全覆盖 / 部分覆盖 / 未覆盖 |
| 匹配度 | 高 / 中 / 低 |
| 对应内容位置 | 文档中的具体位置 |
| 优化建议 | 如何提升覆盖度和匹配度 |

**统计输出**：
- 总体覆盖度 %
- 未覆盖评分项（含分值和改进建议）
- 覆盖不足评分项

**建议**：此维度强烈建议提供 `--reference` 参考文件（招标分析报告），以便准确对照评分标准。

### 2.4 风险识别（Risk Check）

**配置文件**：`prompts/risk_check.yaml`

**检查逻辑**：按 4 大风险分类识别文档中的潜在风险。

**风险分类**：

| 分类 | 说明 | 典型风险 |
|------|------|---------|
| 废标风险 | 可能导致投标无效 | 实质性要求未响应、格式不符 |
| 合规风险 | 法律/法规/标准违规 | 资质缺失、承诺超出能力 |
| 履约风险 | 执行阶段可能的问题 | 工期不合理、技术方案可行性 |
| 争议风险 | 可能引发争议或纠纷 | 条款模糊、权责不清 |

**v1.3.0 新增**：
- 并行写作章节一致性风险检查（针对 taw v1.8.0+）：
  - H3 子节间术语统一性
  - 相邻 H3 过渡自然度
  - 同一技术描述跨 H3 一致性
  - 评分项覆盖完整性

## 3. 工作流程（5 Phase）

### Phase 0：参数解析与初始化

```
0.0 参数解析
    ├── 帮助参数 → 输出帮助退出
    ├── 待审核文件路径（必选）
    ├── --type（必选）: tender_doc/analysis/outline/chapter/full_bid
    ├── --reference（可选）: 参考文件
    ├── --level（可选，默认 all）: critical/high/all
    ├── --focus（可选）: completeness/compliance/scoring/risk
    ├── --revise-docx（可选）: 审核完成后自动生成修订版 DOCX
    └── --revise-scope（可选，默认 must）: must/all

0.05 配置加载（读取 trv.default_level）
0.1 输出目录初始化
0.2 文件读取（待审核文件 + 参考文件）
0.3 启动确认
```

### Phase 1：多维度审核执行

根据 `--focus` 决定执行的维度：

- 未指定 `--focus` → 执行全部 4 个维度（1.1 → 1.2 → 1.3 → 1.4）
- 指定 `--focus` → 仅执行指定维度

**v1.4.0 分块审核模式**（full_bid/chapter 超过 80K token 阈值时自动启用）：

```
1.0.1 上下文模式决策
    ├── 估算内容 token 量（DOCX 用文本字符数 × 1.5，Markdown 用字节数 / 2）
    ├── < 80K tokens → inline 内联模式（原有流程）
    └── ≥ 80K tokens → chunked 分块模式：
        1.0.2 Phase A：结构提取 + 动态分块规划
              ├── 读取标书目录，提取 L2/L3 章节
              ├── 估算各章节体量
              ├── 动态分块：L2 章节 > 3万字 → 下沉到 L3
              ├── chapter 分块：按 H3 子节拆分；无 H3 时按每 800 行强制切分
              └── 定位招标文件关键区域（废标条款/评分标准/技术要求）
        1.0.3 Phase B：按 Chunk 分发子 Agent 审核（每批 ≤ 4 个并行）
              ├── 每个 chunk 跑 4 维度 + 提取关键数据点
              └── 降级守卫：>50% 失败 → 暂停后续批次
        1.0.4 Phase C：跨 Chunk 一致性检查（专用 Agent）
              ├── 数值一致性（同一参数跨章节是否矛盾）
              ├── 描述一致性（架构/组件描述是否矛盾）
              ├── 承诺一致性（时间/SLA 是否自洽）
              └── 全局完整性（目录对应、格式统一）
        1.0.5 Phase D：分块结果汇总 → 进入 Phase 2
```

**内联模式执行流程**（不变）：
1. 读取对应的 Prompt 配置文件
2. 根据 `--type` 选择检查清单/审查要点
3. 执行检查/审查/分析
4. 按 `--level` 过滤结果
5. 生成 Markdown 格式的检查结果

### Phase 2：审核结果汇总

```
2.1 生成审核结论
    ├── 统计问题总数和严重程度分布
    ├── 评估总体风险等级（高/中/低）
    └── 判断总体评价：
        ├── 通过：无严重问题，一般问题 ≤ 3
        ├── 有条件通过：无严重问题，一般问题 > 3
        └── 不通过：存在严重问题

2.2 生成问题清单（按严重程度分类 + 修改建议）

2.3 生成修改优先级
    ├── 必须修改（阻塞性问题）
    ├── 建议修改（重要问题）
    └── 可选优化
```

### Phase 3：Markdown 报告生成

整合 Phase 1 和 Phase 2 的所有结果，生成结构化报告：

```markdown
# 审核报告 - <审核类型>
## 基本信息
## 审核结论
## 详细审核结果（1.1-1.4）
## 问题清单
## 修改优先级
## 附录（检查清单引用 + 法规引用清单）
```

**输出文件**：`output/trv/审核报告_<type>_<timestamp>.md`

### Phase 3.5：DOCX 修订版生成

启用 `--revise-docx` 时，在 Markdown 审核报告生成后执行自动修订：

1. 检查输入是否为 `.docx`
2. 检查审核类型是否为 `outline` / `chapter` / `full_bid`
3. Claude 基于审核报告动态生成修订指令 JSON
4. 调用 `tools/trv_docx_reviser.py --instructions <json>` 执行修订
5. 调用 `tools/docx_encoding_check.py --fix` 执行修订后编码检查
6. 将修订结果（applied/not_found/errors）写入终端输出和报告附录

**输出文件**：`output/trv/<原文件名>_修订版_<timestamp>.docx`

**降级规则**：
- 非 DOCX 输入：跳过修订，不中断审核
- 不支持的审核类型：跳过修订，不中断审核
- 修订失败：保留审核报告，提示人工处理

### Phase 4：任务完成与清理

输出确认 + 后续建议（根据审核类型和结果提供针对性建议）。

## 4. 动态检查逻辑

v1.1.0 起，outline 和 full_bid 的检查项**不再是固定章节名列表**，改为基于大纲实际内容的动态关系检查。

**设计原因**：taa v2.1.0+ 大纲结构发生重大变化（仅保留技术部分，编号从 5.X 改为 1.X），固定检查清单会失效。

**动态检查示例**：
- "大纲中是否有响应招标文件中所有【必须】技术要求的章节" （而非固定检查 "1.3 总体方案设计" 是否存在）
- "各章节间是否存在内容重叠或逻辑矛盾" （基于实际内容语义分析）

## 6. Prompt 文件架构

```
skills/trv/
├── SKILL.md                              # 主编排 Prompt（含分块审核编排逻辑）
└── prompts/
    ├── completeness.yaml                 # 完整性检查清单（按审核类型分节，含 scope 标记）
    ├── compliance.yaml                   # 合规性审查要点（含法规引用，含 scope 标记）
    ├── scoring_alignment.yaml            # 评分契合度分析模板（含 scope 标记）
    ├── risk_check.yaml                   # 风险识别清单（4 大分类，含 scope 标记）
    ├── fullbid_chapter_agent.yaml        # [v1.4.0] 章节级审核子 Agent 提示词模板
    └── fullbid_consistency_agent.yaml    # [v1.4.0] 跨 chunk 一致性检查 Agent 模板
```

**scope 标记说明**（v1.4.0）：

full_bid 检查项新增 `scope` 字段，用于分块模式下的检查项分配：
- `scope: chapter` — 章节级检查，由 Phase B 子 Agent 在每个 chunk 内执行
- `scope: global` — 全局级检查，由 Phase C 一致性检查 Agent 执行（如全文数据一致性、目录对应、页码连续）

## 7. 与其他 Skill 的关系

```
[TPL] 技术规格与评标办法 ──→ TRV --type tender_doc
[TAA] 招标分析报告 ──→ TRV --type analysis
[TAA] 投标文件大纲 ──→ TRV --type outline（建议 --reference 分析报告）
[TAW] 章节草稿 ──→ TRV --type chapter（建议 --reference 分析报告）
[TAW] 合并标书 ──→ TRV --type full_bid（建议 --reference 招标文件）
```

TRV 的审核结果可指导 TPL/TAA/TAW 修改：
- 审核不通过 → 重新执行对应 Skill
- 有条件通过 → 修改问题后可继续
- 通过 → 进入下一流程

**完整工作流**：
```
甲方: 需求 → [TPL] → [TRV] 审核 → 发布招标文件
乙方: 招标文件 → [TAA] → [TRV] 审核 → [TAW] → [TRV] 审核 → 提交投标
```
