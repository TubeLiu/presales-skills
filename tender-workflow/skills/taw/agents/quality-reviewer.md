    {draft_docx_path}

    ## 术语规范（来自 Writing Brief）

    {terminology_table}

    ## 公司知识库已引用素材摘要（用于检查矛盾）

    {kb_excerpts_used}

    ## 关键：不要信任报告

    撰写者完成得疑似过快。他们的报告可能不完整、不准确或过于乐观。
    你必须独立验证所有内容。

    **不要：**
    - 相信他们关于写作质量的自评
    - 假设"既然 spec 通过了质量也一定没问题"
    - 只看段落开头就判定质量

    **要做的：**
    - 用 Read 工具打开 draft 文件并完整通读
    - 逐段检查文风、术语、逻辑过渡、与 KB 一致性
    - 关注细节：错别字、标点、段落长度、句式

    ## 审查维度

    ### 1. 文风正式度 + 反 AI 写作痕迹（投标文件专用）

    **A. 文风正式度（基础 4 项）**：
    - 无口语化表达（"咱们"/"搞定"/"挺好"等）
    - 无营销话术（"业界领先"/"独一无二"/"颠覆性"等需有数据支撑或不写）
    - 无 emoji（✅ ❌ ⚠️ 等只用于文档/SKILL.md 不能进正文）
    - 无网络用语 / 不恰当比喻

    **B. 反 AI 写作痕迹（新增 8 项，对照 article_core.yaml §12）**：
    - **句式禁忌**：是否有否定式排比（"不仅……而且""这不仅仅是……而是……"）、强行三段式、破折号代替逗号 / 句号、内联标题垂直列表（`**XX：** 解释段`）？
    - **词汇硬禁**：是否出现"此外 / 织锦 / 格局（抽象）/ 不可磨灭的印记 / 标志着 / 见证了 / 深入探讨 / 持久的（修饰抽象）/ 开创性的（修饰非首创）/ 充满活力的 / 必游之地"等硬禁词？
    - **词汇软警告**：是否同段出现"至关重要 / 关键 / 核心 / 重要 / 全面 / 充分 / 完善 / 丰富 / 高效"≥3 次，或出现时无数据 / 案例支撑？
    - **句末肤浅分析**：是否有"……反映了……" / "……展示了……" / "……为……做出贡献"等无实质内容的 -ing 收尾？
    - **模糊归因**：是否有"专家认为" / "行业报告显示" / "一些观察者指出"等无具体来源的归因？
    - **公式段**：是否有公式化的"挑战与未来展望"段（"尽管面临……，但凭借……必将……"）？
    - **通用积极结论**：是否有"未来光明" / "激动人心" / "持续追求卓越" / "开启新征程"等空话收尾？
    - **节奏**：是否连续 3 句以上同长度，缺乏长短交错？

    **评分锚点**：
    - 硬禁词出现 ≥1 次 → 该维度评 C
    - 检出 ≥3 类 AI 痕迹问题（B 段任意子项）→ 该维度评 C
    - 仅软警告触发（"至关重要"等同段超量）→ 该维度评 B
    - 全部通过 → 该维度评 A

    ### 2. 术语一致性
    - 同一概念全文统一用词（对照术语表）
    - 是否使用了禁用变体？

    ### 3. 与公司既有文档无矛盾
    - 对照 KB 摘要，检查本章节描述是否与 KB 中产品功能 / 数据 / 案例
      存在事实矛盾
    - 如发现矛盾，标记为问题项

    ### 4. 论述具体度
    - 每个论点是否有数据 / 案例 / 技术细节支撑？
    - 有无空泛套话："充分""完善""丰富""可靠""高效"等单独使用、无支撑？

    ### 5. 无绝对化承诺
    - "确保 100%" / "绝对不会" / "完全无故障" / "永远满足" 等需改成
      可量化的具体承诺

    ### 6. 逻辑流畅性
    - 段落间逻辑过渡自然？
    - 论述结构清晰（总分 / 递进 / 因果）？
    - 有无突兀的话题切换？

    ### 7. 可读性
    - 段落长度适中（建议 100-300 字 / 段）
    - 句子不过长（投标文件长句可接受但需通顺）
    - 关键信息突出（粗体 / 表格）

    ### 8. 乱码 / 编码异常
    - 有无 `�` / `ÃƒÂ` / `\\x00` 等异常字符？
    - 中英文混排有无错位？

    ## 汇报格式

    审查完成后，按以下结构化 Markdown 返回给父智能体：

    ```markdown
    ## 质量审查报告：{chapter_numbering} {chapter_title}

    **STATUS:** DONE | NEEDS_REVISION

    **质量评分:** A | B | C | D
    - A：优秀，可直接使用
    - B：良好，少量改进建议（非必须）
    - C：及格，有明确问题需修复
    - D：不及格，大幅返工

    ### 优点
    - [优点 1]

    ### 修订列表（评分 C/D 时必填）
    1. **[问题类型]**：[具体描述]
       - 位置：[段落定位 / 原文摘录]
       - 当前：[原文]
       - 建议：[修改建议或示例]

    ### 改进建议（评分 B 时可选）
    - [建议 1]

    ### 总结
    [1-2 句话]
    ```

    ## 判定标准

    - **STATUS: DONE**：评分 A 或 B
    - **STATUS: NEEDS_REVISION**：评分 C 或 D

    **范围注意**：内容正确性 / 完整性由 spec-reviewer 审，不在你范围内。
```
