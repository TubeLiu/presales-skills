# Tender Planner — 招标策划助手

> 一条命令，从产品功能清单到无控标痕迹的招标技术规格与评标办法。

**Tender Planner** 是一个 **Claude Code** AI Skill，专为甲方招标团队打造。它能将乙方产品功能清单自动转换为甲方视角的招标技术规格与评标办法，核心能力是**反控标转换**——去除品牌痕迹、中性化术语、合理化指标，生成自然、合规的招标文件。

## 为什么需要它

| 传统做法 | 使用 Tender Planner |
|---------|-------------------|
| 手动改写产品功能描述，容易留下控标痕迹 | AI 自动执行反控标转换，品牌残留零容忍 |
| 凭经验判断指标合理性，容易被质疑 | 指标合理化（下限表达，浮动 70-85%） |
| 评标办法与技术规格脱节 | 从技术规格自动映射评分维度，交叉检查 |
| 不同行业要求不同，难以兼顾 | 4 种行业模板（政府/金融/央国企/通用企业） |
| 大型项目和小型项目用同一套模板 | 4 级细致程度（detailed/standard/general/brief） |

## 核心亮点

**反控标转换（核心能力）** — 4 步转换流程：术语中性化（去品牌名）→ 指标合理化（下限表达）→ 需求泛化（功能结果导向）→ 分布均衡（多域覆盖），并执行自动自检报告。

**控标点强制纳入** — 产品功能描述中标注"控标点"的条目，不受 level 压缩比影响，保留核心技术指标，优先映射为高分值评分项。

**行业模板差异化** — 4 种行业模板自动注入专项章节：政府（等保/信创）、金融（两地三中心/监管合规）、央国企（自主可控/国产化）、通用企业（灵活部署/快速交付）。

**评标办法自动生成** — 从技术规格自动映射技术评分维度，量化要求→客观评分项，描述性要求→主观评分项，并执行必须项↔评分项交叉检查。

## 使用方法

```bash
# 基本用法（政府行业，标准细致程度）
/tpl features.txt --template government

# 含项目背景，金融行业，详细输出
/tpl features.txt --project overview.txt --template finance --level detailed

# 使用知识库产品索引
/tpl --kb --template soe --level brief

# 文件+知识库组合输入，只生成技术规格
/tpl features.txt --kb --template enterprise --no-scoring

# 查看帮助
/tpl -h
```

### 参数说明

| 参数 | 类型 | 说明 |
|------|------|------|
| `<产品功能清单>` | 位置参数 | 必选（或 `--kb`）。产品功能文件（.txt/.md/.xlsx/.pdf） |
| `--kb` | 标志 | 可选。使用知识库产品索引，可与文件输入组合 |
| `--project <file>` | 文件路径 | 可选。项目概述（背景、预算、规模） |
| `--template <行业>` | 字符串 | 必选。行业类型（见下表） |
| `--level <级别>` | 字符串 | 可选，默认 `standard`。细致程度（见下表） |
| `--no-scoring` | 标志 | 可选。跳过评标办法，只生成技术规格 |
| `-h, --help` | 标志 | 可选。显示帮助信息 |

### 行业模板

| 模板 | 适用场景 | 行业专项章节 | 技术权重 |
|------|---------|------------|---------|
| `government` | 政府机关、事业单位 | 等保安全要求、国产化/信创要求 | 50% |
| `finance` | 银行、保险、证券 | 金融级高可用、金融级安全、监管合规 | 50% |
| `soe` | 中央企业、国有企业 | 自主可控技术路线、国产化适配、长期技术支持 | 45% |
| `enterprise` | 民营企业、外资企业 | 灵活部署、快速交付、成本优化 | 40% |

### 细致程度

| 级别 | 技术规格页数 | 功能要求条目 | 适用场景 |
|------|------------|------------|---------|
| `detailed` | 15-20 页 | 40-60 条 | 大型项目 >500 万 |
| `standard`（默认） | 8-12 页 | 15-25 条 | 中型项目 100-500 万 |
| `general` | 4-6 页 | 8-12 条 | 小型项目 50-100 万 |
| `brief` | 2-3 页 | 5-8 条 | <50 万或内部参考 |

## 工作流程

```
Phase 0  参数解析 → 读取产品功能清单（文件/KB/组合）→ 加载行业模板
Phase 1  功能分析 → 识别控标点 → 提取量化指标 → 构建功能注册表
Phase 2  反控标转换（核心）→ 术语中性化 → 指标合理化 → 需求泛化 → 分布均衡 → 自检
Phase 3  技术规格生成 → 按 level 控制深度 → 注入行业专项章节 → 一致性检查
Phase 4  评标办法生成 → 从技术规格映射评分维度 → 交叉检查（除非 --no-scoring）
Phase 5  DOCX 输出 → 反控标最终检查 → 文件交付
```

输出文件保存在 `./output/tpl/` 目录（CLI 环境）。

## 输出文件

| 模式 | 文件名 |
|------|--------|
| 含评标办法（默认） | `技术规格与评标办法_<项目名>_<时间戳>.docx` |
| 仅技术规格（`--no-scoring`） | `技术规格_<项目名>_<时间戳>.docx` |

文档结构：封面 → 目录 → 第一部分：技术规格与要求 → 第二部分：评标办法（若有）

## 与 trv 集成

tpl 生成后，建议通过 trv 进行深度审核：

```bash
# Step 1: 生成技术规格与评标办法
/tpl features.txt --template government

# Step 2: 审核（反控标合规性 + 评标办法合规性 + 内部一致性）
/trv output/tpl/技术规格与评标办法_项目名_20260313.docx --type tender_doc
```

**分工**：
- **tpl 的反控标自检**（生成时）：品牌残留检查、分布集中度检查、要求等级比例检查
- **trv 的深度审核**（审核时）：反控标合规性深度审查、技术评分与技术规格对应关系检查、内部一致性风险识别

## 项目结构

```
tpl/
├── README.md                    # 本文件
├── SKILL.md                     # Claude Code Skill 主文件（Phase 0-5 完整 prompt）
├── prompts/
│   ├── anti_control.yaml        # 反控标转换规则（核心）
│   ├── level_rules.yaml         # 四级细致程度定义
│   ├── technical.yaml           # 技术规格编写规则
│   └── scoring.yaml             # 评标办法设计规则
└── templates/
    ├── README.md                # 模板使用指南（详细）
    ├── government.yaml          # 政府行业模板
    ├── finance.yaml             # 金融行业模板
    ├── soe.yaml                 # 央国企模板
    └── enterprise.yaml          # 通用企业模板
```
