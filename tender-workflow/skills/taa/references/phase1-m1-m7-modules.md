## 1. Phase 1 执行步骤

### Step 1: 准备上下文

1. 使用 Read 工具读取 `prompts/analysis.yaml`
2. 将文件内容中的所有 `{VENDOR_NAME}` 替换为实际的厂商名（在内存中完成）
3. 产品能力上下文准备（按优先级选择来源）：
   - 若 `--product` 提供 且 `KB_SOURCE_OVERRIDE != "anythingllm"` → 使用 `PRODUCT_INDEX`（最高优先，精确评估）
   - 否则若 `PRODUCT_SOURCE=anythingllm` → M2 阶段对每条技术要求按需调用 `anythingllm_search` 查询（见下方 §2）
   - 否则若 `PRODUCT_INDEX` 存在（本地 V2/V1 索引）→ 将其完整内容添加到上下文
   - 否则 → AI 模糊评估（无产品能力数据）

### Step 2: 加载行业模板（如果需要）

1. 根据 Phase 0.2 判断的行业类型，使用 Read 工具读取对应的行业模板文件
2. 将模板文件内容中的所有 `{VENDOR_NAME}` 替换为实际的厂商名
3. 准备在 M1 之后插入行业扩展模块

### Step 2.5: 标记映射预扫描

在执行 M1-M7 分析前，先对招标文件进行全文扫描，建立特殊标记的语义映射表：

1. 扫描全文，识别所有特殊标记的定义语句（如"带有★号的条款为实质性要求"、"标注*的为加分项"、"△为可选项"等）
2. 建立标记映射表（标记符号 → 语义含义 → 定义出处）
3. 若文档中未找到显式定义，采用行业默认映射：★ = 必须/废标，* = 重要，△ = 可选
4. 输出标记映射表供后续 M2/M5 使用：

   ```
   标记映射表：
   • ★ = 实质性要求，不满足按废标处理（出处：第一章 投标人须知 P.5）
   • * = 加分项（出处：第三章 评分标准 P.22）
   • 无其他标记定义
   ```

### Step 3: 执行7模块分析

按照 `analysis.yaml` 中定义的框架，结合行业扩展模块和标记映射表，生成完整分析报告。注意：M2 分析须使用标记映射表判定优先级，M7 最后须执行 7.6 跨模块交叉验证，验证结果若发现遗漏须回溯补充 M2/M5 表格。

## 2. M2 AnythingLLM 查询规则

仅当 `PRODUCT_SOURCE=anythingllm` 时执行。

**查询缓存机制**：

- 初始化空字典 `QUERY_CACHE = {}`
- 对每个技术要求条目，先提取规范化查询键（去除空格、转小写）
- 若查询键已在 `QUERY_CACHE` 中 → 直接使用缓存结果，不重复调用
- 缓存命中率输出日志：`"AnythingLLM 查询：命中缓存 [N] 条，新查询 [M] 条"`

**查询执行流程**（对每条新查询）：

```
1. 构建查询关键词：
   - 从技术要求原文提取核心关键词（技术术语、功能名称、指标名称）
   - 组合查询语句："<功能名称> <技术参数>"

2. 执行查询：
   anythingllm_search(
     query="<查询语句>",
     workspace=ANYTHINGLLM_WS
   )

3. 结果处理：
   - 若查询成功：
     - 取 score >= 0.65 的最高分结果
     - 提取产品能力描述用于支持度评估
     - 记录查询日志："[查询] <关键词> → 匹配成功 (score=0.XX)"
   - 若查询失败（工具报错）：
     - 记录失败日志："[失败] <关键词> → AnythingLLM 查询失败"
     - 降级标记：该条目标记为"AI 模糊评估（查询失败）"
     - 继续处理下一条，不中断整体流程
   - 若无结果（score < 0.65）：
     - 记录无匹配日志："[无匹配] <关键词> → 知识库无匹配结果"
     - 降级标记：该条目标记为"AI 模糊评估（无匹配）"

4. 支持度判定：
   - 有匹配结果（score >= 0.65）→ 基于产品能力描述判定支持度
   - 无匹配结果 → 标注"⚠️ 知识库无匹配，AI 模糊评估"
```

**查询日志输出示例**：

```
AnythingLLM 查询进度：
• T1 多集群管理：✅ 匹配成功 (score=0.82) → 完全支持
• T2 容器镜像仓库：✅ 匹配成功 (score=0.71) → 需定制
• T3 服务网格：⚠️ 无匹配 (score<0.65) → AI 模糊评估
• T4 国产化适配：✅ 匹配成功 (score=0.91) → 完全支持
总计：查询 25 条，匹配 18 条，无匹配 7 条，缓存命中 3 条
```

**限制**：

- 每条技术要求最多调用 1 次 AnythingLLM，避免过多 API 调用
- 单次查询超时时间：30 秒
- 最大并发查询数：1（串行执行，避免速率限制）

## 4. 行业扩展模块（按行业加载）

**政府IT 扩展**（插入 M1 之后）：

从 `templates/builtin/government_it.yaml` 加载以下三个模块：

- **E1. 预算合规分析**：预算金额合理性、资金来源合规性、最高限价、付款方式
- **E2. 政府采购合规分析**：采购方式合规、中小企业政策、进口产品限制、时间要求
- **E3. 安全保密分析**：等保级别要求、数据本地化、安全产品国产化要求

**国企/央企 扩展**（插入 M1 之后）：

从 `templates/builtin/state_owned_enterprise.yaml` 加载以下三个模块：

- **E1. 国资监管合规分析**：采购管理合规、投资决策程序、信息公开要求
- **E2. 信创国产化要求分析**：CPU/OS/数据库/中间件具体型号要求、信创认证
- **E3. 廉政风险防控分析**：廉洁从业要求、回避制度、审计监督条款

**金融扩展**（插入 M1 之后）：

从 `templates/builtin/finance.yaml` 加载以下三个模块：

- **E1. 金融监管合规分析**：银保监会/证监会/央行监管要求
- **E2. 数据安全与隐私保护分析**：金融数据分级、个人金融信息保护、访问控制
- **E3. 业务连续性分析**：RTO/RPO 要求、高可用架构、灾备体系

## 5. 分析质量要求

- 每条分析标注原文出处（章节/页码），格式：`（出处：第X章 P.XX）`
- 废标项标注 `[废标]`，扣分项标注 `[扣分]`，难点标注 `[难点]`
- 必须条款标注 `【必须】`，废标条款标注 `【必须/废标】`
- 信息缺失时填"**文件未明确**"，禁止臆测数字
- M7 撰写指导必须针对本项目具体展开，禁止通用模板内容
- M2 技术能力类须使用参考分类 + 动态扩展模式（不局限于 9 个预定义分类），服务管理类 6 个固定分类须全部覆盖（无对应要求的注明"本项目未涉及"）
- M2 提取须覆盖全格式（表格/段落/列表/附件），不得因格式差异遗漏
- M5 废标检测须覆盖全文档所有章节，包括附件和合同范本
- M7 须执行 7.6 跨模块交叉验证，验证结果若发现遗漏须立即回溯补充 M2/M5

## 7. Phase 1 交接文件 phase1_handoff.yaml 规范

在保存 Markdown 报告文件的同时，生成结构化交接文件 `phase1_handoff.yaml`，供 Phase 2 使用。此文件从刚生成的 M1-M7 分析结果中提取结构化数据，避免 Phase 2 重新解析完整报告。

**生成步骤**：

1. 创建临时目录：

   ```bash
   mkdir -p ./output/.temp
   ```

2. 使用 Bash 执行 Python 脚本，将以下数据写入 `./output/.temp/phase1_handoff.yaml`：

**交接文件内容规范**：

```yaml
# phase1_handoff.yaml — Phase 1 → Phase 2 结构化交接
generated_at: "[ISO 时间戳]"
report_file: "[Phase 1 报告文件的完整路径]"

# ── 项目基本信息（来自 M1）──
project_name: "[项目名称]"
vendor_name: "[厂商名称]"
industry_type: "[行业类型: government_it | finance | state_owned_enterprise | general]"

# ── M2 技术要求完整条目（保留原文，确保 Part C 精度）──
m2_requirements:
  - id: "T1"
    category: "[需求分类名]"
    requirement: "[技术要求完整原文]"
    source: "[出处：第X章 P.XX]"
    priority: "[【必须】或【必须/废标】或 普通]"
    marker: "[★ 或 * 或 空]"
    support_status: "[完全支持 | 需定制 | 风险-需评估 | 不支持]"
  # ... 列出 M2 中所有条目（不仅限于【必须】，全部列出）

# ── M2 分类汇总（用于大纲结构决策）──
m2_categories:
  - name: "[分类名]"
    type: "technical"          # technical | service_management
    must_count: 8              # 该分类下【必须】条目数
    total_count: 12            # 该分类下总条目数
    has_star: true             # 是否含★号条目

# ── M4 评分要点（≥10分项完整，其余摘要）──
m4_scoring:
  - id: "S1"
    item: "[评分项名称]"
    max_score: 15
    category: "[技术方案 | 实施服务 | 安全合规 | ...]"
    strategy: "[得分策略简述]"

# ── M5 废标与风险（完整列出）──
m5_risks:
  - id: "R1"
    type: "[废标 | 扣分 | 风险]"
    clause: "[条款简述]"
    source: "[出处]"

# ── M7 写作优先级 ──
m7_priorities:
  - priority: 1
    task: "[优先撰写任务]"
    related_scoring: "[关联评分项]"

# ── 行业扩展模块摘要（若有）──
industry_extensions:
  - module_id: "E1"
    title: "[模块标题]"
    key_findings: "[关键发现摘要，1-2句]"

# ── 标记映射表 ──
marker_mapping:
  - marker: "★"
    meaning: "[实质性要求/废标]"
    source: "[定义出处]"
```

**关键要求**：

- M2 条目必须保留**完整原文**和**出处引用**，不可做摘要压缩（Phase 2 Part C 需要原文精度）
- M2 列出所有条目，不仅限于【必须】（Part C 和大纲结构需要完整视图）
- 若 Phase 1.5 匹配验证更新了 M2 条目，交接文件须反映更新后的结果
- 交接文件生成失败不影响 Phase 1 报告的正常输出（非阻塞）

## 8. Phase 1.5 匹配验证

**执行位置**：在 Phase 1 完成、保存 Markdown 文件之前执行。

**执行条件**：仅当用户提供了 `--product` 参数且 PRODUCT_INDEX 存在时执行。

对 M2 技术要求矩阵中标记为 `⚠️` 的条目进行二次验证，减少误报。

### 验证流程

**步骤一**：收集 M2 表格中所有支持度为 `⚠️[需定制]` 或 `⚠️[风险-需评估]` 的条目。

**步骤二**：对每条执行扩展检索：

1. **同义词扩展**：扩展关键词搜索（如"集群"↔"K8s"、"容器"↔"Docker"、"高可用"↔"HA"）
2. **跨分类检索**：检查其他分类下是否有相关条目
3. **模糊匹配**：检查描述字段是否包含相关语义

**步骤三**：更新匹配结果：

```
验证更新：
• T5 多集群管理：⚠️[风险-需评估] → ✅完全支持（匹配依据：技术能力清单_3）
• T12 容器镜像仓库：⚠️[需定制] → 保持（说明：需确认是否支持 Harbor 对接）
...
验证后误报减少：[N] 条
```

### 输出更新

若验证阶段有更新，修改 M2 表格对应条目：

- 更新支持度
- 更新匹配依据（填写 entry.id）
- 在备注中添加"（验证后更新）"标记
